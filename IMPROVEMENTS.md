# Code Review — Areas of Improvement

A comprehensive review of the Files Manager codebase identifying issues grouped by severity and category.

---

## Critical / Security

### 1. Local storage upload/download endpoints are unauthenticated

**File:** `apps/backend/src/modules/storage/local-storage.controller.ts:47-48`

Both `upload` and `download` endpoints are marked `@Public()`, meaning anyone can upload or download files without authentication. While the signed-URL pattern works for GCS (where the URL itself is the auth), the local storage controller has no equivalent protection.

**Impact:** Any unauthenticated user can read or write files if they guess the object path.

**Solution:** Remove `@Public()` from both endpoints and validate the JWT token. Alternatively, add a short-lived token mechanism (e.g., a query-string token generated by the backend that the local-storage controller validates), similar to how GCS signed URLs work.

```ts
// Instead of @Public(), validate a short-lived token
@Put('upload/*')
async upload(@Req() req: Request, @Res() res: Response) {
  const token = req.headers.authorization?.replace('Bearer ', '');
  // Validate token or use a dedicated upload token from the upload-request flow
  // ...
}
```

### 2. Path traversal risk in local storage controller

**File:** `apps/backend/src/modules/storage/local-storage.controller.ts:35-43`

The path normalization logic strips leading `..` sequences but the regex `replace(/^(\.\.(\/|\\|$))+/, '')` only handles leading traversal. A path like `foo/../../etc/passwd` would pass the check after `path.normalize()` resolves it to `../etc/passwd`, but the subsequent `startsWith` check would catch this. However, the logic is fragile and hard to audit.

**Impact:** Potential directory traversal if the normalization logic has edge cases.

**Solution:** Simplify to a single, well-tested check — resolve the path and verify it's within the storage root:

```ts
const target = path.resolve(this.localRootAbs(), objectPath);
const root = this.localRootAbs();
if (!target.startsWith(root + path.sep)) {
  throw new BadRequestException('Invalid path');
}
```

Remove the custom regex normalization and rely on `path.resolve()` + `startsWith()` alone.

### 3. `.env` file committed with real secrets

**File:** `apps/backend/.env`

The `.env` file contains default but real-looking credentials (`DATABASE_PASSWORD=postgres`, `JWT_SECRET=change-me`, `ADMIN_PASSWORD=AdminPass123!`). While `.gitignore` has `**/.env`, the file is present on disk with actual values. The `ADMIN_PASSWORD` default in `seed-admin.ts` is hardcoded.

**Impact:** If `.gitignore` fails or the file is accidentally committed, secrets are exposed.

**Solution:**
- Ensure the `.env` file is never committed (verify with `git status`).
- Change the default `JWT_SECRET` in `app.config.ts` from `'change-me'` to throw an error if not set in production.
- In `seed-admin.ts`, require `ADMIN_PASSWORD` to be set explicitly rather than having a default.

```ts
// app.config.ts
jwt: {
  secret: process.env.JWT_SECRET ?? (process.env.NODE_ENV === 'production'
    ? (() => { throw new Error('JWT_SECRET must be set in production'); })()
    : 'dev-only-secret'),
},
```

### 4. No rate limiting on auth endpoints

**File:** `apps/backend/src/modules/auth/auth.controller.ts`

The `register` and `login` endpoints have no rate limiting, making them vulnerable to brute-force and credential-stuffing attacks.

**Impact:** Attackers can attempt unlimited logins.

**Solution:** Add rate limiting using `@nestjs/throttler`:

```ts
// app.module.ts
import { ThrottlerModule, ThrottlerGuard } from '@nestjs/throttler';

@Module({
  imports: [
    ThrottlerModule.forRoot([{ ttl: 60000, limit: 10 }]),
    // ...
  ],
  providers: [
    { provide: APP_GUARD, useClass: ThrottlerGuard },
    // Keep existing JwtAuthGuard...
  ],
})
```

Apply stricter limits specifically to auth routes.

### 5. JWT token stored in accessible cookie

**File:** `apps/frontend/app/composables/useAuth.ts:18`

The JWT is stored via `useCookie('access_token')`, which by default is not `httpOnly`. This makes it accessible to JavaScript (XSS exfiltration).

**Impact:** If an XSS vulnerability exists anywhere in the app, the token can be stolen.

**Solution:** Move to an httpOnly cookie set by the backend on login. The backend should set `Set-Cookie` with `httpOnly`, `secure`, and `SameSite=Strict` flags. The frontend would then rely on cookies being sent automatically.

## High Priority

### 6. Duplicate permission-check logic in FilesService

**File:** `apps/backend/src/modules/files/files.service.ts:82-149`

The `assertCanRead`, `assertCanWrite`, and `assertCanDelete` methods are nearly identical — they differ only in the `canRead`/`canWrite`/`canDelete` column and the error message. This violates DRY and makes maintenance error-prone.

**Solution:** Extract a single generic method:

```ts
private async assertPermission(
  folderId: string,
  groupNames: string[],
  userId: string,
  permission: 'canRead' | 'canWrite' | 'canDelete',
) {
  if (groupNames.includes('Admin')) return;
  const folder = await this.foldersRepository.findOneBy({ id: folderId });
  if (folder?.createdById === userId) return;
  if (!groupNames.length) {
    throw new ForbiddenException(`No ${permission.replace('can', '').toLowerCase()} access for this folder.`);
  }
  const count = await this.permissionsRepository
    .createQueryBuilder('permission')
    .innerJoin('permission.group', 'group')
    .where('permission.folderId = :folderId', { folderId })
    .andWhere('group.name IN (:...groupNames)', { groupNames })
    .andWhere(`permission.${permission} = true`)
    .getCount();
  if (!count) {
    throw new ForbiddenException(`No ${permission.replace('can', '').toLowerCase()} access for this folder.`);
  }
}
```

### 7. No error handling for GCS signed URL generation

**File:** `apps/backend/src/modules/storage/adapters/gcs-storage.adapter.ts`

The GCS adapter does not handle errors from `getSignedUrl` or `delete`. If the bucket doesn't exist, credentials are invalid, or GCS is down, the error will propagate as an unhandled 500.

**Solution:** Wrap GCS calls with proper error handling:

```ts
async getUploadUrl(objectPath: string, mimeType: string): Promise<string> {
  try {
    const [signedUrl] = await this.storage.bucket(this.bucketName).file(objectPath).getSignedUrl({
      version: 'v4',
      action: 'write',
      expires: Date.now() + 15 * 60 * 1000,
      contentType: mimeType,
    });
    return signedUrl;
  } catch (error) {
    throw new InternalServerErrorException('Failed to generate upload URL');
  }
}
```

### 8. FileEntity uses `bigint` for size but DTO uses `number`

**File:** `apps/backend/src/modules/files/entities/file.entity.ts:26` and `apps/backend/src/modules/files/dto/upload-request.dto.ts:14`

The entity declares `size` as `bigint`, but the DTO uses `@IsInt()` / `Min(1)` with JavaScript `number`. This creates a type mismatch — JavaScript `number` cannot safely represent values beyond `2^53`. Additionally, TypeORM returns `bigint` as a string by default, which can cause serialization issues.

**Solution:**
- If files are expected to be under 2^53 bytes (which is ~9 PB), use `integer` in the entity instead of `bigint`:
  ```ts
  @Column({ type: 'integer' })
  size!: number;
  ```
- If true bigint support is needed, handle string-to-number conversion in the service and use a custom transformer.

### 9. No file ownership or upload verification

**File:** `apps/backend/src/modules/files/files.service.ts:43-54`

The `create` method registers file metadata after a client uploads to a signed URL, but there is no verification that the upload actually happened. A client could call `POST /folders/:id/files` with any path without having uploaded anything.

**Solution:** Add an upload verification step:
- For GCS: check that the object exists in the bucket before creating the metadata record.
- For local storage: verify the file exists on disk before creating the record.
- Add a `createdById` field to `FileEntity` to track who uploaded the file.

### 10. No pagination on list endpoints

**Files:** `files.controller.ts`, `folders.controller.ts`, `groups.controller.ts`

All list endpoints return every record with no pagination. As data grows, these endpoints will become slow and memory-intensive.

**Solution:** Add pagination with `skip`/`take` or cursor-based pagination:

```ts
@Get()
list(
  @Query('page') page = 1,
  @Query('limit') limit = 50,
  @Param('folderId', ParseUUIDPipe) folderId: string,
  @CurrentUser() user: AuthUser,
) {
  return this.filesService.list(folderId, user.groups, user.sub, { page, limit });
}
```

### 11. `synchronize: true` is dangerous for production data

**File:** `apps/backend/src/database/typeorm.config.ts:16`

While the code guards with `process.env.NODE_ENV !== 'production'`, synchronize can still accidentally drop columns or tables in development, destroying data. It also makes schema changes implicit and hard to track.

**Solution:** Use TypeORM migrations instead:
- Set `synchronize: false` everywhere.
- Generate migration files with `typeorm migration:generate`.
- Run migrations on startup or via CI.

### 12. Folders list query for non-admin users is inefficient

**File:** `apps/backend/src/modules/folders/folders.service.ts:17-53`

The `listForUser` method runs 2-3 separate queries and then a final query with `In([...idSet])`. This is inefficient and doesn't scale well.

**Solution:** Use a single query with a `UNION` or subquery:

```ts
async listForUser(groupNames: string[], userId: string) {
  if (groupNames.includes('Admin')) {
    return this.foldersRepository.find({
      order: { createdAt: 'DESC' },
      relations: ['permissions', 'permissions.group'],
    });
  }

  return this.foldersRepository
    .createQueryBuilder('folder')
    .leftJoinAndSelect('folder.permissions', 'permission')
    .leftJoinAndSelect('permission.group', 'group')
    .where('folder.createdById = :userId', { userId })
    .orWhere((qb) => {
      const subQuery = qb
        .subQuery()
        .select('perm.folderId')
        .from(FolderPermission, 'perm')
        .innerJoin('perm.group', 'grp')
        .where('grp.name IN (:...groupNames)', { groupNames })
        .andWhere('perm.canRead = true')
        .getQuery();
      return `folder.id IN ${subQuery}`;
    })
    .orderBy('folder.createdAt', 'DESC')
    .getMany();
}
```

## Medium Priority

### 13. No tests for business logic

**Files:** All service files

The only tests are the scaffolded `app.controller.spec.ts` and `app.e2e-spec.ts`, which test the "Hello World" endpoint. There are zero tests for auth, files, folders, groups, or storage logic.

**Solution:** Write unit tests for each service, starting with the most critical:
1. `AuthService` — registration, login, duplicate email, wrong password
2. `FilesService` — permission checks, upload validation, file CRUD
3. `FoldersService` — permission logic, ownership checks
4. `StorageService` — adapter selection, signed URL generation

### 14. `AppController` and `AppService` are scaffold leftovers

**Files:** `apps/backend/src/app.controller.ts`, `apps/backend/src/app.service.ts`

These only return "Hello World!" and serve no purpose in the application.

**Solution:** Remove both files and the `AppController`/`AppService` references from `AppModule`.

### 15. Inconsistent env variable naming

**Files:** `apps/backend/.env.example` vs `apps/backend/src/config/app.config.ts`

The `.env.example` uses `DATABASE_URL` but `typeorm.config.ts` reads individual `DATABASE_HOST`, `DATABASE_PORT`, etc. The example is misleading.

**Solution:** Either:
- Remove `DATABASE_URL` from `.env.example` and keep individual vars, or
- Parse `DATABASE_URL` in the TypeORM config (useful for platforms like Heroku/Render that provide a connection string).

### 16. Frontend API errors are silently swallowed

**File:** `apps/frontend/app/plugins/api-client.ts`

The `$apiFetch` plugin does not handle HTTP errors centrally. Each composable has its own try/catch, but many just show a generic toast. API error responses (validation errors, 403 details) are lost.

**Solution:** Add a response interceptor:

```ts
const apiFetch = async <T>(url: string, options = {}) => {
  try {
    return await $fetch<T>(url, { baseURL: config.public.apiBaseUrl, ...options, headers });
  } catch (error: any) {
    if (error?.data?.message) {
      throw new Error(error.data.message);
    }
    throw error;
  }
};
```

### 17. Frontend auth middleware does not protect admin routes

**File:** `apps/frontend/app/middleware/auth.ts`

The middleware checks for a token but does not check admin status. The admin pages (`/admin/folders`, `/admin/groups`) do their own client-side check, but this happens after the page loads — there is a brief flash where the admin UI could be visible.

**Solution:** Add admin-specific middleware:

```ts
// middleware/admin.ts
export default defineNuxtRouteMiddleware(async (to) => {
  const { user, fetchMe } = useAuth();
  if (!user.value) await fetchMe();
  if (!user.value?.groups?.includes('Admin')) {
    return navigateTo('/folders');
  }
});
```

### 18. `FileStatus` enum only has one value

**File:** `apps/backend/src/modules/files/entities/file-status.enum.ts`

The enum has a single member `READY`. This suggests the status field was designed for a multi-state lifecycle (e.g., UPLOADING, PROCESSING, READY, ERROR) but was never implemented.

**Solution:** Either implement the full lifecycle or remove the enum and column until it's needed. If keeping it, add states like `PENDING` and `UPLOADING` to reflect the actual upload flow.

### 19. Upload validation reads env vars directly instead of using ConfigService

**File:** `apps/backend/src/modules/files/files.service.ts:159-162`

```ts
const maxSize = imageMimes.includes(mimeType)
  ? Number(process.env.UPLOAD_MAX_IMAGE_BYTES ?? 25 * 1024 * 1024)
  : Number(process.env.UPLOAD_MAX_PDF_BYTES ?? 50 * 1024 * 1024);
```

This bypasses the `ConfigService` and the validated config in `app.config.ts`.

**Solution:** Inject `ConfigService` and read from the validated config:

```ts
const imageMax = this.configService.get<number>('uploadPolicy.imageMaxBytes');
const pdfMax = this.configService.get<number>('uploadPolicy.pdfMaxBytes');
```

### 20. No transaction on folder creation with permissions

**File:** `apps/backend/src/modules/folders/folders.service.ts:56-81`

The folder and its permissions are saved in separate operations. If the permission save fails, you end up with a folder that has no permissions.

**Solution:** Wrap in a transaction:

```ts
async create(createdById: string, dto: CreateFolderDto) {
  return this.foldersRepository.manager.transaction(async (manager) => {
    const folder = manager.create(Folder, { name: dto.name, createdById });
    await manager.save(folder);

    if (dto.permissions?.length) {
      const perms = dto.permissions.map(p =>
        manager.create(FolderPermission, { folderId: folder.id, ...p }),
      );
      await manager.save(perms);
    }

    return manager.findOne(Folder, {
      where: { id: folder.id },
      relations: ['permissions', 'permissions.group'],
    });
  });
}
```

## Low Priority / Code Quality

### 21. No request logging or observability

The backend has no request logging, health check endpoint, or structured logging. In production, debugging issues will be difficult.

**Solution:**
- Add `Logger` from NestJS to each service.
- Add a health check endpoint (`@Public() @Get('health')`).
- Consider integrating a logging library (Pino, Winston) for structured logs.

### 22. CORS configuration could be stricter

**File:** `apps/backend/src/main.ts:7-15`

CORS is enabled with `credentials: true`. The `CORS_ORIGIN` env var allows any comma-separated origins. There's no validation.

**Solution:** Validate CORS origins at startup and reject invalid values.

### 23. Frontend uses `confirm()` for delete

**File:** `apps/frontend/app/pages/folders/index.vue:149`

Using `window.confirm()` breaks the UI consistency and is blocked in some browser contexts (iframes).

**Solution:** Create a custom confirmation modal component.

### 24. File size displayed as raw bytes

**File:** `apps/frontend/app/components/file-manager/ItemGrid.vue:71`

`{{ file.size }} bytes` shows raw byte count (e.g., "26214400 bytes"). This is not user-friendly.

**Solution:** Add a utility to format bytes:

```ts
export function formatBytes(bytes: number): string {
  if (bytes < 1024) return `${bytes} B`;
  if (bytes < 1024 * 1024) return `${(bytes / 1024).toFixed(1)} KB`;
  if (bytes < 1024 * 1024 * 1024) return `${(bytes / (1024 * 1024)).toFixed(1)} MB`;
  return `${(bytes / (1024 * 1024 * 1024)).toFixed(1)} GB`;
}
```

### 25. Groups controller `list()` is not guarded

**File:** `apps/backend/src/modules/groups/groups.controller.ts:11`

The `list()` endpoint returns all groups to any authenticated user. This may leak group structure.

**Solution:** If groups should be admin-only, add `@UseGuards(AdminGuard)`. If groups should be visible to all users, this is fine — but document the decision.

### 26. No `updatedAt` timestamps on entities

**Files:** `file.entity.ts`, `folder.entity.ts`, `group.entity.ts`

Entities only have `createdAt`. There is no way to track when a record was last modified.

**Solution:** Add `@UpdateDateColumn()` to entities that can be modified.

### 27. Frontend `.output` directory is committed

The `.output/` directory (Nuxt build output) is present in the repo. The `.gitignore` in `apps/frontend/` should catch it, but the root `.gitignore` pattern `apps/frontend/.output/` needs to be verified.

**Solution:** Ensure `.output/` is properly ignored and remove tracked build artifacts from git history.

### 28. Docker builds from workspace root context

**File:** `bitbucket-pipelines.yml:37-38`

The pipeline runs `docker build -f apps/backend/Dockerfile apps/backend`, which means the Docker context is the backend directory only. The backend Dockerfile does `COPY . .` which copies the entire backend directory — this is fine, but if any workspace-root configuration is needed, it won't be available.

**Solution:** This is acceptable for the current setup, but document the Docker context assumption.

### 29. No input sanitization on folder/file names

**Files:** `files.service.ts`, `folders.service.ts`

Folder and file names are not sanitized. A user could create a folder named `<script>alert(1)</script>` which, when rendered in the frontend, could cause XSS.

**Solution:** Either:
- Sanitize on the backend (strip HTML tags), or
- Ensure the frontend always uses `v-text` or equivalent (Vue's `{{ }}` does auto-escape, so this is partially safe already).

Vue's template interpolation does auto-escape, so this is low risk — but explicit sanitization is still a best practice.

## Architecture / Scalability

### 30. Monolithic permission checks in service layer

Permission checks are scattered across `FilesService` and `FoldersService` with inline queries. As the app grows, maintaining consistency will be hard.

**Solution:** Extract a `PermissionsService` or use NestJS CASL integration for a centralized authorization layer.

### 31. No database indexing strategy beyond the unique constraint

**File:** `apps/backend/src/modules/folders/entities/folder-permission.entity.ts:13`

The only index is the unique compound on `(folderId, groupId)`. Queries filtering by `folderId` alone or `groupId` alone will do sequential scans.

**Solution:** Add indexes on:
- `files.folderId` (for listing files by folder)
- `files.status` (for filtering ready files)
- `folders.createdById` (for owned-folder queries)

### 32. No file versioning or soft delete

Files are hard-deleted from both the database and storage. There's no way to recover accidentally deleted files.

**Solution:** Implement soft delete (`deletedAt` column + `ARCHIVED` status) and consider versioning for file updates.
